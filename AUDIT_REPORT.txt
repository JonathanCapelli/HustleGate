🔴 PROBLÈMES CRITIQUES (à corriger avant publication)
================================================================================

───────────────────────────────────────────────────────────────────────────────
CRITIQUE #1 : R8/ProGuard désactivé en release
Fichier : app/build.gradle.kts (ligne 22)
───────────────────────────────────────────────────────────────────────────────

  Code actuel :
      release {
          isMinifyEnabled = false
      }

  Problèmes :
  - L'APK/AAB de release ne sera PAS obfusqué → le code est décompilable
  - L'APK sera ~50% plus gros que nécessaire (ressources non minifiées)
  - Google Play exige de plus en plus un bundle optimisé

  Correction recommandée :
      release {
          isMinifyEnabled = true
          isShrinkResources = true
          proguardFiles(
              getDefaultProguardFile("proguard-android-optimize.txt"),
              "proguard-rules.pro"
          )
      }

  ⚠️ Nécessite de créer un fichier proguard-rules.pro avec les règles pour :
  - ML Kit Pose Detection
  - Google Play Billing
  - Health Connect
  - AdMob (déjà géré par le SDK)

───────────────────────────────────────────────────────────────────────────────
CRITIQUE #2 : Pas de gestion du cycle de vie BillingClient
Fichier : BillingManager.kt
───────────────────────────────────────────────────────────────────────────────

  Problème :
  - BillingClient n'est jamais déconnecté (pas de endConnection())
  - Si l'Activity est détruite pendant un achat, le callback
    onPurchaseComplete capture une référence morte → fuite mémoire potentielle
  - Pas de reconnexion automatique si le service se déconnecte

  Impact : Fuite mémoire + achats potentiellement non traités

  Correction recommandée :
  - Ajouter une méthode endConnection() appelée dans onDestroy
  - Gérer la reconnexion dans onBillingServiceDisconnected()
  - Nettoyer onPurchaseComplete quand l'Activity se détruit

───────────────────────────────────────────────────────────────────────────────
CRITIQUE #3 : CameraProvider jamais libéré explicitement
Fichier : CameraExerciseScreen.kt (lignes 170-210)
───────────────────────────────────────────────────────────────────────────────

  Problème :
  - La caméra est bindée via cameraProvider.bindToLifecycle() dans un
    AndroidView factory
  - Le DisposableEffect (ligne 145) ferme le poseAnalyzer et le soundFeedback
    mais ne fait PAS cameraProvider.unbindAll()
  - Normalement le lifecycle gère ça, mais l'AndroidView factory crée
    un scope séparé qui pourrait ne pas être nettoyé correctement

  Impact : Caméra restant ouverte, batterie drainée, autres apps ne
  peuvent pas accéder à la caméra

  Correction recommandée :
  - Capturer la référence cameraProvider et la libérer dans DisposableEffect

🟠 PROBLÈMES IMPORTANTS (fortement recommandé de corriger)
================================================================================

───────────────────────────────────────────────────────────────────────────────
IMPORTANT #1 : Pas de feedback utilisateur si la caméra échoue
Fichier : CameraExerciseScreen.kt (lignes 200-205)
───────────────────────────────────────────────────────────────────────────────

  Code actuel :
      } catch (e: Exception) {
          Log.e("CameraExercise", "Camera init failed", e)
      }

  Problème : Si la caméra échoue (utilisée par une autre app, hardware
  défaillant), l'utilisateur voit un écran noir sans aucun message.

  Correction : Ajouter un état d'erreur et afficher un message + bouton retry.

───────────────────────────────────────────────────────────────────────────────
IMPORTANT #2 : Double appel potentiel de checkAndRewardSteps
Fichier : HomeScreen.kt (lignes 105-130)
───────────────────────────────────────────────────────────────────────────────

  Il y a deux LaunchedEffect qui peuvent appeler checkAndRewardSteps() :
  1. LaunchedEffect(Unit) - boucle polling toutes les 1 sec (NE l'appelle PAS
     directement, OK)
  2. LaunchedEffect(healthAvailable) - appelle checkAndRewardSteps()
  3. healthPermissionLauncher callback - appelle aussi checkAndRewardSteps()

  La protection hasStepRewardToday() dans HealthConnectManager empêche le
  double crédit, mais il y a un appel réseau inutile à Health Connect à
  chaque recomposition si healthAvailable change.

  Impact : Mineur, mais gaspillage de ressources.

───────────────────────────────────────────────────────────────────────────────
IMPORTANT #3 : Écriture SharedPreferences sur chaque frappe (Settings)
Fichier : SettingsScreen.kt
───────────────────────────────────────────────────────────────────────────────

  Chaque modification d'un TextField (reps, minutes, objectif pas) écrit
  immédiatement dans SharedPreferences. Pas de debounce.

  Impact : Micro-lag sur appareils anciens pendant la saisie clavier.
  Pas critique mais pas idéal.

───────────────────────────────────────────────────────────────────────────────
IMPORTANT #4 : formatTime() dupliqué dans 3 fichiers
Fichiers : HomeScreen.kt, CameraExerciseScreen.kt, TimerWidgetProvider.kt
───────────────────────────────────────────────────────────────────────────────

  La même fonction formatTime(millis: Long) est copiée dans 3 fichiers.
  Un bug corrigé dans l'un ne sera pas corrigé dans les autres.

  Correction : Extraire dans un fichier utilitaire (ex: TimeUtils.kt).

───────────────────────────────────────────────────────────────────────────────
IMPORTANT #5 : StatsManager stocke le JSON dans SharedPreferences
Fichier : StatsManager.kt
───────────────────────────────────────────────────────────────────────────────

  L'historique complet des exercices est stocké en JSON dans une seule
  clé SharedPreferences. Après des mois d'utilisation intensive (>1000
  exercices), la lecture/écriture pourrait devenir lente car tout le
  JSON est rechargé à chaque appel.

  Impact : Performance dégradée à long terme.
  
  Amélioration future : Migrer vers Room (SQLite) si nécessaire. Pour
  une v1, c'est acceptable.

───────────────────────────────────────────────────────────────────────────────
IMPORTANT #6 : Streak cassé si l'utilisateur change de fuseau horaire
Fichier : StatsManager.kt (updateStreak, lignes 70-82)
───────────────────────────────────────────────────────────────────────────────

  Le streak compare des dates LocalDate.now().toString() mais si l'utilisateur
  voyage et change de fuseau horaire, "aujourd'hui" peut devenir "hier" ou
  "demain" et casser la logique du streak.

  Impact : Edge case rare. Acceptable pour une v1.

───────────────────────────────────────────────────────────────────────────────
MINEUR #4 : allowBackup=true sans BackupAgent
Fichier : AndroidManifest.xml (ligne 17)
───────────────────────────────────────────────────────────────────────────────

  android:allowBackup="true" est déclaré mais aucun BackupAgent n'est
  configuré. Les SharedPreferences seront sauvegardées par le backup
  automatique Android, ce qui est correct pour notre cas d'usage
  (exercices, réglages). Pas un vrai problème.

───────────────────────────────────────────────────────────────────────────────
MINEUR #5 : Pas de gestion du bouton retour dans BlockedActivity (partiel)
Fichier : BlockedActivity.kt (lignes 70-73)
───────────────────────────────────────────────────────────────────────────────

  Le bouton retour redirige vers le home (goHome()), ce qui est correct.
  Mais les gestes de navigation (swipe) pourraient contourner le blocage
  sur certains launchers.

  Impact : L'utilisateur pourrait swipe-return vers l'app bloquée sur
  certains appareils. L'AccessibilityService re-bloquera mais il y aura
  un flash de l'app.

───────────────────────────────────────────────────────────────────────────────
MINEUR #6 : Pas de crashlytics / analytics
───────────────────────────────────────────────────────────────────────────────

  Pas de Firebase Crashlytics ni d'outil de crash reporting.
  En production, il sera difficile de diagnostiquer les crashs utilisateurs.

  Recommandation : Ajouter Firebase Crashlytics (gratuit) avant la v1.1.
